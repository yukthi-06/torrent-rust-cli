# User Prompt
> a list of announce tracker urls is stored in a txt file trackers.txt. Use this list to create the torrent

# `trackers.txt` Implementation Walkthrough

The CLI and Daemon have been updated to properly parse and embed trackers from `trackers.txt` into generated `.torrent` files in full compliance with BitTorrent BEP 12.

## What changed?

1. **RPC Protocol Extension (`crates/rpc/src/lib.rs`)**
   - The `Request::Create` and `Request::CreateAdd` variants were updated to include an optional list of trackers (`trackers: Option<Vec<String>>`).
   - This architectural change shifts the responsibility of reading the file from the background daemon to the interactive CLI, ensuring the correct working directory is used.

2. **CLI Integration (`crates/cli/src/main.rs`)**
   - When the user runs `torrent create` or `torrent create-add`, the CLI now attempts to read `trackers.txt` (or `crates/trackers.txt`) from its current working directory.
   - It parses the file line-by-line, trims whitespace, and filters out empty lines.
   - The resulting list is packaged into the RPC payload and sent to the daemon.

3. **Torrent Metadata Generation (`crates/daemon/src/creator.rs`)**
   - Refactored `create_torrent` to accept `trackers: Vec<String>`.
   - **Backwards Compatibility:** The first tracker in the list is always mapped to the primary `announce` key.
   - **BEP 12 Compliance:** If multiple trackers are provided, they are mapped to the `announce-list` property. Each tracker is placed in its own independent tier (e.g., `[[url1], [url2]]`) to ensure the torrent client tries them sequentially.

## Verification
You can verify this by running `torrent create-add` and then inspecting the generated `.torrent` file in any standard torrent client (e.g., qBittorrent, Transmission). It will display the full list of 46 trackers from your `trackers.txt` file.
