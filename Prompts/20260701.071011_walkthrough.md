# User Prompt
> once a create-add is done, the file should be served immediately from the location of the file/folder.  But app shows it is downloading....

# In-Place Seeding Implementation Walkthrough

The `torrent create-add` command will now accurately identify the original path of the created torrent and begin instantly seeding it, instead of attempting to download it into the global download directory!

## What changed?

1. **State Persistence Upgrade (`crates/daemon/src/server.rs`)**
   - Introduced a new `SavedTorrent` struct to replace the naive `(u32, String)` tuple that we previously stored in `torrents.json`.
   - The `SavedTorrent` struct now explicitly tracks:
     - `id`: The internal torrent ID.
     - `path`: The path to the `.torrent` file or the `magnet:?` link.
     - `download_dir`: An optional custom directory path!
   - Upgraded `load_state()` to gracefully try parsing the new `SavedTorrent` schema. If it fails (which it will on the first boot for your existing torrents), it seamlessly falls back to reading the old tuple format and migrating them in-memory so you don't lose any data!

2. **Dynamic Download Directory Dispatch**
   - Refactored `add_torrent_internal` to accept a `custom_download_dir: Option<PathBuf>`.
   - Updated `Request::CreateAdd` so that when a torrent is built, the parent directory of the original target (e.g. `C:\path\to` for `C:\path\to\my_folder`) is passed into `add_torrent_internal` as the overriding download directory.
   - When the daemon starts up or a torrent is manually resumed, `restore_torrent` pulls the custom directory out of the `saved_paths` cache. If it exists, it uses it instead of falling back to the global `self.config.download_dir`.

## Validation

Try it out by running:
```powershell
cargo check
cargo build
```

Then create a new torrent via the CLI:
```powershell
torrent create-add C:\path\to\my_folder
```

When you run `torrent list`, it should skip the downloading phase and immediately transition from "Checking" to "Seeding"!
