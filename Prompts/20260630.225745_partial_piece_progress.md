# User Prompt
> on restart, 58% complete became 47% complete.
> Surely data is getting writen every 5 seconds, we shouldnt lose 11% of data for a 1GB file....

# Implementation
- Discovered that the drop in percentage on restart is inherently caused by BitTorrent's block-by-block downloading vs piece-by-piece verification.
- The daemon was tracking block-level progress in `downloaded` bytes, inflating the percentage before full pieces were verified. When the daemon restarts, any partially-downloaded piece fails the cryptographic hash check and is completely discarded.
- Fixed a major flaw where disconnected peers abandoned their partially downloaded pieces forever, leading to permanently lost pieces and permanently inflated progress stats.
- Added a `PieceGuard` which detects if a peer disconnects midway through downloading a piece. The guard now automatically refunds the partial downloaded bytes back from the `downloaded` total, and re-queues the piece into a thread-safe `missing_pieces` queue so another peer can securely finish it.
- Fixed a bug where the final piece's size was miscalculated, leaving the final peer connection hanging infinitely.
