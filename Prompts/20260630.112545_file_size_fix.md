# User Prompt
> one of the torrent downloaded by this app is around 1000 bytes more than the file size of the same torrent downloaded by Deluge torrent client.
> Analyze and fix it

# Bug Fix: File Over-Allocation in Torrents

## Issue
Torrents were occasionally downloading around a kilobyte larger than they should. This happened because the client was always requesting exactly `16384` bytes (16KB blocks) when fetching pieces from peers, regardless of whether it was the very last block in the torrent.

For the final block in a torrent, the actual remaining bytes are usually less than 16KB. Since the application always requested 16KB and blindly wrote whatever was returned, the final file on disk was padded with garbage or null bytes up to the full 16KB size, inflating the file size compared to the actual torrent metadata length.

## Fix Implemented
1. **Dynamic Block Requests**: Modified `handle_peer` in `crates/daemon/src/engine.rs` to compute the exact requested block size. A new closure `get_request_length` calculates the minimum of `16384`, the remaining bytes in the piece, and the remaining bytes in the entire torrent. 
2. **Bounds Checking for Single Files**: Updated `write_block` for `FileMode::Single` to explicitly check and truncate the block data if it tries to write past the total `length` of the file (similar to the boundary checks already present in `FileMode::Multi`).

This ensures files are now saved with byte-for-byte exactness to the torrent's metadata size, matching standard clients like Deluge.
