# User Prompt
> the one 100% completed torrent is now shoing this in log
> 
> Torrent XYZ.mkv: 385/870 pieces verified, resuming from piece 385

# Implementation
- Identified a bug where `TorrentDownloader::handle_peer` would redundantly assign the exact same `first_missing` starting piece to *every single connected peer*. 
- This caused the same pieces to be downloaded repeatedly, artificially inflating the `downloaded` count and prematurely marking the torrent as "Completed" before all data was actually downloaded to disk.
- Implemented basic piece picking logic by adding `next_piece_to_request: AtomicU32` to `TorrentDownloader`.
- Each peer now fetches a unique piece index atomically, ensuring no two peers download the same piece, and accurately representing `downloaded` bytes.
