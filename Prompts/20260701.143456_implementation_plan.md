# Seeding Implementation Plan

Currently, the daemon operates exclusively as a **leecher**. It connects to peers to download pieces but ignores incoming piece requests, doesn't listen for incoming connections, and doesn't broadcast when it finishes a piece. 

To turn the daemon into a fully compliant BitTorrent seeder, we will implement the following architectural changes:

## User Review Required
> [!IMPORTANT]
> The daemon will now listen on TCP port `6881` by default for incoming peer connections. Please ensure this port is open in your firewall if you plan to seed publicly.

## Proposed Changes

### 1. Global Peer Listener (`crates/daemon/src/server.rs`)
- **[MODIFY]** `server.rs`: Add a background task during server initialization that binds a `tokio::net::TcpListener` to `0.0.0.0:6881`.
- **[MODIFY]** `server.rs`: When a peer connects, read their `Handshake`. Look up their requested `info_hash` across our active torrents. If we have it, dispatch the TCP stream to that torrent's `handle_peer` logic.
- **[MODIFY]** `server.rs`: Update `TorrentHandle` to store an `Arc<engine::TorrentDownloader>` so the server can easily dispatch incoming connections to the correct engine instance.

### 2. Piece Broadcasting (`crates/daemon/src/engine.rs`)
- **[MODIFY]** `engine.rs`: Add a `tokio::sync::broadcast::Sender<u32>` to `TorrentDownloader`.
- **[MODIFY]** `engine.rs`: Whenever the downloader successfully verifies and writes a new piece to disk, it will broadcast the piece index. All active peer connections will subscribe to this broadcast and immediately send a `PeerMessage::Have` to their remote peer, notifying the swarm that we have new data available.

### 3. Bidirectional Peer Handling (`crates/daemon/src/engine.rs`)
- **[MODIFY]** `engine.rs` (`handle_peer`): Split the `TcpStream` into concurrent read/write halves using `stream.into_split()`.
- **[MODIFY]** `engine.rs` (`handle_peer`): 
  - **Bitfield**: Immediately upon connection, calculate and send our current `Bitfield` representing all pieces we currently possess.
  - **Unchoking**: When the remote peer sends an `Interested` message, we will reply with an `Unchoke` message so they can begin requesting pieces.
  - **Serving Pieces**: When the remote peer sends a `Request(index, begin, length)` message, we will read the data directly from the disk using `read_piece_from_disk()` and dispatch a `Piece` message back to them.
  - **Metrics**: Track uploaded byte counts in `TorrentState::uploaded` so the CLI reflects our seeding contribution!

## Verification Plan
1. **Compilation**: Run `cargo check` and `cargo build` to ensure the asynchronous split-stream refactor compiles flawlessly.
2. **Local Seeding Test**: Create a torrent, start seeding it in the daemon, and open the same `.torrent` file in a commercial client (like qBittorrent). We will observe qBittorrent successfully downloading pieces from our daemon over port `6881` and see our CLI's `Uploaded` counter increase!
