# Implement `torrent create-add` functionality

This plan outlines the steps required to implement the `torrent create-add <path>` command, which will create a new torrent and immediately add it to the daemon to start serving it.

## Proposed Changes

### Shared Protocol (RPC)
#### [MODIFY] [lib.rs](file:///X:/Projects_X/0_Active/E_Rust/torrent_rust_cli_GITHUB/crates/rpc/src/lib.rs)
- Add `CreateAdd { path: String }` variant to the `Request` enum so the CLI can transmit this new command to the daemon.

### CLI Crate
#### [MODIFY] [main.rs](file:///X:/Projects_X/0_Active/E_Rust/torrent_rust_cli_GITHUB/crates/cli/src/main.rs)
- Add the `CreateAdd` command to the Clap-based `Commands` enum.
- In the command matching block, route the CLI command to `Request::CreateAdd` and dispatch it to the daemon.

### Daemon Crate
#### [MODIFY] [server.rs](file:///X:/Projects_X/0_Active/E_Rust/torrent_rust_cli_GITHUB/crates/daemon/src/server.rs)
- Refactor the existing logic for `Request::Add` into a new helper method: `pub async fn add_torrent_internal(self: &Arc<Self>, mut path_or_magnet: String) -> Response`. This allows both `Request::Add` and `Request::CreateAdd` to use the exact same logic.
- Update `Request::Add` to call `add_torrent_internal`.
- Implement `Request::CreateAdd` to:
  1. Offload the torrent hashing/creation to `tokio::task::spawn_blocking` (just like `Request::Create`).
  2. Write the `.torrent` file to disk.
  3. Instead of returning `Response::Info`, immediately execute `self.add_torrent_internal(out_path).await` and return the result to the CLI.

## Verification Plan
### Manual Verification
- Run `cargo build`.
- Execute `torrent create-add <some-path>`.
- Verify the CLI output indicates the torrent was added successfully with a new ID.
- Run `torrent list` to verify the daemon is now "Checking" or "Seeding" the newly created torrent.
