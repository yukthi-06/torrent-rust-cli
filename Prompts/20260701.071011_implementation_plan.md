# Implement In-Place Seeding for `create-add`

When using `torrent create-add C:\path\to\my_folder`, the daemon was reverting to using the global `download_dir` defined in the config. Since `my_folder` is not in the global download directory, the daemon attempts to download it from scratch instead of seeding it from its original location!

To fix this, we need to allow the daemon to track a custom `download_dir` for each torrent and persist that custom path across daemon restarts.

## Proposed Changes

### Daemon Crate

#### [MODIFY] [server.rs](file:///X:/Projects_X/0_Active/E_Rust/torrent_rust_cli_GITHUB/crates/daemon/src/server.rs)
- **State Migration**: Introduce a `SavedTorrent` struct to replace the simple `(u32, String)` tuple in `torrents.json`:
  ```rust
  #[derive(Serialize, Deserialize, Clone)]
  struct SavedTorrent {
      id: u32,
      path: String,
      download_dir: Option<String>,
  }
  ```
- Update `load_state()` to attempt to parse the new `SavedTorrent` struct. If it fails (due to the old format), it will gracefully fall back to parsing the old tuple format and seamlessly migrate it.
- **Internal RPC Changes**: Update `add_torrent_internal` to accept a `custom_download_dir: Option<String>` parameter. 
- When `Request::CreateAdd` is invoked, it will compute the parent directory of the target file/folder (e.g., `C:\path\to`) and pass it as the custom download directory.
- `restore_torrent` will be updated to query the `saved_paths` hash map. If a custom download directory is defined for the torrent, it will use that instead of the global `self.config.download_dir`.
