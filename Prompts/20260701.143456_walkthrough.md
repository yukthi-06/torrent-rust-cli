# Seeding Implementation Walkthrough

The `torrentd` daemon has been successfully upgraded from a simple leecher client into a full-fledged BitTorrent seeder capable of sharing downloaded pieces with the global swarm!

## Changes Made

### 1. The Global Peer Listener
A background TCP listener was added to `crates/daemon/src/server.rs` that binds to port `6881` when the daemon starts.
When an incoming connection is made, the daemon:
- Reads the BitTorrent handshake packet to extract the `info_hash`
- Locates the active `TorrentDownloader` handling that specific torrent
- Dispatches the raw TCP socket to the downloader's new `handle_incoming_peer` function

### 2. Live Piece Broadcasting
To ensure we can alert connected peers the millisecond we finish downloading and verifying a new piece, a `tokio::sync::broadcast` channel was added to the `TorrentDownloader` state. 

### 3. Bidirectional Peer Stream Split
The entire `handle_peer` architecture in `crates/daemon/src/engine.rs` was refactored using `stream.into_split()`. This safely separates reading and writing halves of the socket so we can use a `tokio::select!` block to seamlessly:
1. Listen for new messages from the peer (like `Request` for a piece, or `Piece` for a block we need)
2. Listen for internal broadcast events signaling that we finished a piece so we can instantly transmit a `Have` message to the remote peer.

### 4. Uploading and Unchoking Logic
- When a remote peer connects, we immediately transmit a calculated `Bitfield` array representing all pieces we currently possess.
- If the remote peer responds with an `Interested` flag, we reply with an `Unchoke` packet, signaling they have clearance to request pieces.
- If they transmit a `Request` block, we safely validate the index, synchronously read the specific byte offset from disk using `read_piece_from_disk()`, and transmit the raw bytes back as a `Piece` message!

### 5. Uploaded Metrics Tracking
The `TorrentState` now correctly tracks the total bytes uploaded, which is serialized and sent to the CLI during `torrent list` and `torrent status` queries!

## How to Verify

1. Run `cargo build` in your workspace.
2. Start the daemon locally. It will bind to port `6881`.
3. Start a commercial client (like qBittorrent or Transmission) on another machine or isolated instance.
4. Add the *exact same* `.torrent` file to the commercial client and force the commercial client to add your local daemon IP and port `6881` as a peer.
5. In your command line, run `torrent list` and `torrent status`. You will see the `Uploaded` metric start climbing as your daemon serves blocks to the commercial client!

> [!WARNING] 
> Seeding requires an open port! If you plan to test this over the internet rather than a local LAN, ensure you've port-forwarded `6881` (TCP) in your router settings.
